# v2.2.1 — SBOM Re-Anchor Patch (operational, no code changes)

**Release date:** 2026-04-30
**Tag:** `v2.2.1` (framework). `@kullailabs/mcop-core@0.2.0` and `mcop@3.2.0` registry versions remain unchanged.

This is an **operational patch only**. There are no functional changes from
[v2.2.0](./v2.2.0.md). v2.2.1 exists solely to re-anchor the v2.2.0 release
on a fresh GitHub Release page that has the CycloneDX SBOMs attached as
downloadable assets.

## Why this release exists

During the v2.2.0 release sequence:

1. The Python package `mcop@3.2.0` was published to PyPI successfully via
   the refactored OIDC Trusted-Publishing flow.
2. The post-publish step that attaches CycloneDX SBOMs to the GitHub
   Release page (`softprops/action-gh-release@v2.6.2`) failed with:

   ```
   Cannot upload asset mcop-core.cdx.json to an immutable release.
   GitHub only allows asset uploads before a release is published, so
   upload assets to a draft release before you publish it.
   ```

3. We refactored `publish-pypi.yml` (#559) to a draft-then-publish flow so
   future releases attach SBOMs at creation time, before the
   immutability lock takes effect.
4. To recover the existing v2.2.0 Release, we deleted it and tried to
   re-create the Release with SBOMs attached at creation. GitHub's API
   rejected the re-creation:

   ```
   422 Validation Failed
   {"resource": "Release", "code": "custom", "field": "tag_name",
    "message": "tag_name was used by an immutable release"}
   ```

   This block is permanent — once a tag has been used by an immutable
   release, no future release object can ever re-bind to that tag, even
   after the original release is deleted.

5. v2.2.1 is the resulting workaround: a fresh tag (`v2.2.1`) bound to a
   no-op CHANGELOG-only commit, with both SBOMs attached at Release
   creation time using the draft → publish flow.

## What's actually attached

- `mcop-framework.cdx.json` — CycloneDX 1.7 SBOM for the framework
  workspace, generated by `pnpm sbom` on the v2.2.0 commit (`5ff1b32…`).
- `mcop-core.cdx.json` — CycloneDX 1.7 SBOM for `@kullailabs/mcop-core`,
  same generation step.

Both are byte-identical to the SBOMs that would have been attached to
v2.2.0 had the immutability collision not occurred. They are validated
against the official CycloneDX JSON schemas via `pnpm sbom:validate`.

## What's not in this release

No code changes. No registry version bumps. No re-publishing of any
artefact to npm or PyPI. The only repository diff between the v2.2.0
commit (`5ff1b32…`) and the v2.2.1 commit is this file plus the v2.2.1
CHANGELOG entry.

## What downstream consumers should do

- For installable artefacts: continue using `mcop@3.2.0` from PyPI and
  `@kullailabs/mcop-core@0.2.0` (currently still pending publication;
  the npm-side Trusted Publisher fix is queued separately) — these are
  the canonical 2026-04-30 framework deliverables.
- For supply-chain auditing: pull the two `*.cdx.json` SBOMs attached
  to the v2.2.1 Release. They describe the v2.2.0 codebase (one commit
  upstream).
- The v2.2.0 tag (`5ff1b32…`) and [release-notes
  document](./v2.2.0.md) remain the canonical reference for what
  shipped in this release cycle.

## Process improvement

The refactored `publish-pypi.yml` (PR #559) now creates the GitHub
Release as a draft with SBOMs attached at creation, then flips
`draft=false` to publish. This sequence is robust against the
Immutable Releases lock and will be applied to `publish-npm.yml` in a
follow-up.
