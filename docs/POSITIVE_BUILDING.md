# Positive Building: “Fixing = Building More Positive Future”

MCOP-Framework-2.0 practices Positive Building: every discovered friction point
is treated as raw material for a more beneficial future. The work is not framed
as subtraction. It is an additive act that leaves the framework more joyful,
more trustworthy, more adoptable, and more powerful at meta-cognitive
optimization.

## Principles

1. **Positive Building of identity resonance** — every public path should guide
   contributors toward the canonical `MCOP-Framework-2.0` home.
2. **Positive Building of deterministic trust** — hashes, timestamps, UUID-v4
   trace IDs, and cosine semantics remain explicit and replayable.
3. **Positive Building of dependency hygiene** — each dependency earns its place
   by increasing clarity, portability, or supply-chain confidence.
4. **Positive Building of doc/code sync** — documentation is a living interface,
   not a decorative layer.
5. **Positive Building of human celebration** — remediation work deserves a
   Merkle-linked memory of the beneficial outcomes it creates.

## The growth ledger

`PositiveResonanceAmplifier` records Positive Building events as append-only,
Merkle-chained growth entries. Each event carries a domain, resonance delta,
evidence payload, and optional human celebration. The resulting metrics are
bounded, deterministic functions of the ledger state:

- contributor joy
- adoption velocity
- beneficial outcome amplification

## The operational loop

Run:

```bash
pnpm positive:audit
```

The command executes the trust suite and emits a Positive Impact Report plus a
badge at `docs/badges/positive-impact.svg`. The badge is intentionally local and
repository-owned, amplifying link integrity while avoiding external score drift.
