# MCTS + MAB + Logically-Learned Rollouts — MCOP Planning Layer

## TL;DR

The planning layer (`src/core/planning/`) augments the deterministic MCOP triad
(NOVA-NEO Encoder, Stigmergy v5, Holographic Etch) with a Monte-Carlo Tree
Search planner whose tree policy is UCB1 (a multi-armed-bandit) and whose
rollouts are **logically learned** — i.e. the simulation phase deterministically
extends the candidate path by greedy resonance maximisation through the triad,
rather than by Monte-Carlo random play-outs.

The planner is orthogonal to the triad: it never mutates encoder, stigmergy, or
etch state during planning. It is opt-in — importing it has zero effect on
existing adapters or pipelines.

## Why

The triad is already a reactive meta-cognitive optimizer (resonance scoring +
incremental confidence updates). The planning layer adds a **proactive** layer
on top:

- **MAB at the Stigmergy boundary** — UCB1 selects between candidate cognitive
  actions for any single decision point.
- **MCTS at the Synthesizer boundary** — UCT (UCB1 inside MCTS) explores
  multi-step orchestration plans before committing.
- **Logically-learned rollouts** — instead of random Monte-Carlo simulation, the
  rollout phase runs a deterministic forward pass through the triad
  (NOVA-NEO encode → Stigmergy resonance query → Holographic Etch
  adaptive-confidence probe) and picks the resonance-maximising next action.
  This preserves crystalline determinism while giving MCTS a much sharper
  evaluation signal than coin-flip play-outs.

## Architecture

```
            ┌──────────────────────────────────────────┐
            │              MCOPMCTSPlanner             │
            │                                          │
   query ──▶│  ┌───────┐  select   ┌──────────────┐    │
            │  │  MAB  │◀──────────│ MCTS tree    │    │
            │  │ UCB1  │  expand   │ (UCT policy) │    │
            │  └───────┘─────────▶ └──────────────┘    │
            │                          │ simulate      │
            │                          ▼               │
            │   ┌────────────────────────────────┐     │
            │   │   Logically-learned rollout    │     │
            │   │   (read-only triad oracle)     │     │
            │   └────────────────────────────────┘     │
            │           │            │           │     │
            │           ▼            ▼           ▼     │
            │      NOVA-NEO     Stigmergy v5   Etch    │
            └──────────────────────────────────────────┘
                              │
                              ▼
                    PlanResult { bestSequence,
                                 rootMerkleHash,
                                 provenanceTrace }
```

## Reward function

For each candidate path tensor `t`:

```
reward(t) = resonance(t) · (1 + adaptiveConfidence(t)) + 0.5 · entropy(t)
```

- `resonance(t)` — `StigmergyV5.getResonance(t).score`, ∈ [0, 1].
- `adaptiveConfidence(t)` — `HolographicEtch.scoreConfidence(t, t).score`,
  ∈ [0, 1]. Uses the existing four-factor adaptive engine
  (alignment, magnitudeHealth, staticFloorMargin, recencyStability).
- `entropy(t)` — `NovaNeoEncoder.estimateEntropy(t)`, ∈ [0, 1]. Included so the
  reward stays discriminative in cold-start regimes (no prior traces).

All three calls are read-only — `scoreConfidence` does not commit to the etch
and `getResonance` does not record a trace. This guarantees the planner cannot
poison the triad with simulation noise.

## Determinism & provenance

- The planner uses no randomness in its rollout; UCB1 is a deterministic tie-
  breaker (untried arms first in caller order, then highest-score arm).
- Every visited node is Merkle-chained to its parent via
  `sha256({ id, action, parentHash, tensorHash })`.
- `PlanResult.rootMerkleHash` is the SHA-256 of all per-node hashes joined by
  `|`, so a single hash uniquely identifies the explored tree shape.
- Node UUIDs differ across runs (they are generated by `randomUUID`), so the
  Merkle chain differs — but `bestSequence` and per-node `meanReward` are
  identical across runs given identical triad state and inputs.

## Usage

```ts
import {
  HolographicEtch,
  MCOPMCTSPlanner,
  NovaNeoEncoder,
  StigmergyV5,
} from '@kuonirad/mcop-framework';

const encoder = new NovaNeoEncoder({ dimensions: 64, normalize: true });
const stigmergy = new StigmergyV5({ resonanceThreshold: 0.3 });
const etch = new HolographicEtch({ confidenceFloor: 0.0 });

const planner = new MCOPMCTSPlanner(
  { encoder, stigmergy, etch },
  { mctsBudget: 200, learnedRolloutDepth: 4, maxFanout: 16, maxDepth: 6 },
);

const plan = planner.plan({
  rootText: 'cinematic dawn over a misty valley',
  actions: ['style:lush', 'style:austere', 'pace:slow', 'pace:rapid'],
});

console.log(plan.bestSequence);     // e.g. ['style:lush', 'pace:slow']
console.log(plan.bestReward);
console.log(plan.rootMerkleHash);   // 64-hex-char Merkle root
console.log(plan.provenanceTrace);  // depth-major list of all visited nodes
```

The planner is fully **opt-in**: existing adapters do not need to use it. After
inspecting the result the caller may choose to commit the plan back to the
triad (e.g. by feeding `bestSequence` to a `BaseAdapter`-derived adapter as
extra prompt context).

## Configuration

`mcop.config.example.json` exposes the same knobs:

```json
"planning": {
  "mctsBudget": 200,
  "ucbC": 1.41421356,
  "learnedRolloutDepth": 4,
  "maxFanout": 16,
  "maxDepth": 6
}
```

## Reversibility

The planning layer is a self-contained subdirectory under `src/core/planning/`
plus a single re-export line in `src/core/index.ts`. Removing the directory
and that re-export reverts the framework to its pre-planning state with no
collateral damage.
